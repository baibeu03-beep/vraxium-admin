## A. 화면에 보이는 영역 · 버튼 (위 → 아래)

**진입 게이트부터.** 목록에서 **공식 휴식** 주차의 [활동 관리]를 누르면 **이동하지 않고** 안내 팝업만 뜹니다("[공식 휴식] 주차로서, 아무런 라인과 그에 수반되는 액트가 체크되지 않습니다."). 공식 활동 주차만 상세로 진입합니다.

| 영역 | 내용 · 버튼 | data 속성 |
|---|---|---|
| **헤더** | 제목 "활동 관리" · 조직 배지(한글명) · "← 주차 내역" 링크 · 도움말 | — |
| **[1] 관리 주차** | "▷ 관리 주차" · 주차명 배지(예 `26 - 여름 - 8`) · 기간 · 공식활동/휴식 배지 · **[주차 검수]** 버튼 · **↩ 실행 취소** · 상태 배지 | `data-managed-week`, `data-review-button` |
| **[2] 현재 주차** | "오늘은, <날짜>" · 시즌 주차명 · 기간 · 클럽 활동 상태 배지 *(빨간 점선 테두리)* | `data-current-week` |
| **[3] 오픈 설정** | "# 이번 주 클럽에서 활동하는 허브와 라인" · **[오픈 확인]** · **[초기화]** · 상태 배지 · "이번 주 활동 인정 개수는 **N**개입니다." | `data-open-confirm-button`, `data-hub-reset-button` |
| **└ 허브 6칸** | (1)[실무 정보] 라인 급(체크) · (2)[실무 정보] 라인(개설) · (3)[실무 경험] 라인 급(체크, 팀×라인급) · (4)[실무 경험] 라인(오픈, 팀×도출·분석·견문·관리·확장) · (5)[실무 역량] 체크/개설(정상 진행) · (6)[클럽 총괄] 라인 급(체크) | `data-hub="info-act"` … `"club-act"` |
| **[4] 하단 탭** | **[액트 체크 관리]** / **[라인 개설 관리]** — 기본 선택 = 액트. 액트=요약 + 허브별 요일 매트릭스(카드 5상태). 라인=요약 + 허브별 7컬럼 표. | `data-tab="act"`/`"line"` |

## B. 정상 완료 흐름 · 선행/완료 조건

통합 관리자(`?org` 없이 `?club`로 진입) 기준. 개별 조직 컨텍스트(`?org`)는 **조회 전용**이라 모든 편집/버튼이 비활성.

1. 목록에서 **종료된(과거) 공식활동 주차**의 [활동 관리] 진입.
2. [1] 관리 주차 · [2] 현재 주차 배너로 대상 주차와 오늘을 구분.
3. 허브 6칸에서 이번 주 실제 활동한 라인급/라인 체크.
   - **선행:** 프로세스 등록에 라인급 존재.
   - **완료:** 화면 체크 상태.
4. **[오픈 확인]** 클릭 → `cluster4_week_opening_configs` 저장 + 인정 개수 N 산정. 과거 주차면 확인 모달 먼저.
   - **완료:** "오픈 설정이 저장되었습니다." 토스트 + 오픈 확인 V.
   - **선행 실패:** 오픈 라인에 포인트 설정 누락 시 **422**(아무것도 저장 안 됨).
5. [액트 체크 관리]·[라인 개설 관리] 탭에서 크루 활동 확인.
6. **[주차 검수]** 클릭 → **검수 준비 상태** 모달(체크리스트, 읽기 전용).
   - **선행:** 오픈 확인 완료(N 확정)·활동 점수 반영·실무 경험 등록/평가·참여 대상자 존재. 하나라도 ❌면 [검수 완료] 버튼 비활성.
7. 모든 항목 ✅이면 모달의 **[검수 완료]** 클릭 → 공표 + 코호트 카드 재계산 + 검수 완료.
   - **완료:** "주차 검수가 완료되었습니다." + 버튼이 "검수 완료"로 고정.
   - **비가역 아님:** ↩ 실행 취소 가능.

## C. 상태별 수정 가능 범위 · 저장 반영처

| 동작 | 권한 · 선행 | 저장/반영 (확인됨) |
|---|---|---|
| **오픈 확인** (POST) | **통합 전용** — `?org` 있거나 단일조직 어드민이면 **403** | `cluster4_week_opening_configs`만(config·open_confirmed·A/B/N). **snapshot·포인트·uws 무접촉.** N은 이후 검수의 성공 기준. |
| **오픈 확인 취소** (DELETE ↩) | 통합 전용 | open_confirmed=false(config 보존) + N null. 멱등. |
| **초기화** | 통합 전용(클라이언트만) | **서버 write 없음.** 화면 체크만 기본값 복원 → 이후 [오픈 확인]으로만 저장. |
| **주차 검수** (POST) | org 접근권만 검사 **(통합 게이트 없음 — A2)**. 준비 모달 통과 필요 | `week_org_result_states`(reviewing→published) · `user_week_statuses` · `weeks.result_published_at/reviewed_at` · 코호트 `weekly_card_snapshots` 재계산 · `user_growth_stats` · 라인 A/B 포인트 정합. **scope=operating/test 독립.** |
| **실행 취소** (DELETE) | reviewed(published)일 때만 버튼 활성 | 공표/검수 해제 + uws 역연산 + 카드 success/fail→집계중 복원. |

### 관련 페이지 반영 · 모드 일치 (검증 완료)

- **주차 내역 목록**의 검수 컬럼과 상세의 상태 배지는 **동일 SoT**(`week_org_result_states` + 동일 scope 해석) → 구조적으로 일치.
- **고객 주차 카드**는 snapshot만 읽음. 일반/`actAsTestUserId`/`demoUserId`는 **읽는 userId만** 다르고 DTO 계산 동일 → 검수가 구운 결과와 일치.
- **operating vs test** 검수 상태·코호트·snapshot은 **독립 격리**. 테스트 검수가 실유저 카드를 건드리지 못함(반대도 동일).
- **인정 개수 N**은 상세·`week-recognitions`·고객 성공/실패 판정이 **공유**. 단 N은 org만 키(모드축 없음).
- **주의:** 오픈 설정 "라인(개설)" 체크박스 ↔ 목록 **라인칸 개설율**은 SoT가 달라(config vs 실제 개설 라인) 토글해도 개설율은 안 변함(설계상, B3).

## R. 결론 · A / B / C

### A · 정책과 무관하게 명백히 수정해야 하는 문제

**A1 · 검수 실패/차단 시 조직 상태가 "주차 검수 중(reviewing)"에 고착**
`markTeamPartsWeekReviewed`가 검증(uws finalize·현재주차 가드)보다 **먼저** `setWeekOrgResultStatus(…, "reviewing")`를 호출하고, 이후 422/에러 시 **aggregating으로 되돌리지 않습니다.** 그 결과 실제 진행이 없는데도 목록·개별 조직 뷰에 "주차 검수 중"이 남습니다. 실행 취소는 published에서만 가능해 개별 뷰는 UI로 복구 불가. **수정:** reviewing은 finalize 성공 직전에 세팅하거나, catch에서 aggregating으로 복원. (테스트 P2-4)

**A2 · 검수·실행취소 라우트에 통합 전용 권한 게이트 없음(오픈 확인과 불일치)**
`open-confirm` 라우트는 `?org 있거나 !isAllOrgs → 403` 게이트가 있으나, `review`의 POST/DELETE는 `guardAdminOrgAccess`(org 접근권)만 검사합니다. UI는 개별(readOnly) 사용자의 검수를 막지만 **서버는 막지 않아**, 개별 컨텍스트에서 직접 호출 시 검수/취소가 통과됩니다. 현재 어드민 전원이 owner라 실피해는 작지만, UI·서버 강제 규칙이 어긋납니다. **수정:** 두 라우트에 동일 게이트 적용(또는 per-org 검수를 정식 허용). (테스트 P3-5)

**A3 · [초기화]가 최초 로드 기본값과 다른 상태를 만듦**
신규 주차 최초 로드는 **실무 역량=체크·액트체크 정보/클럽 라인급=전체 체크**인데, [초기화]는 이들을 **전부 해제**합니다. "초기화"라는 이름과 달리 초기 상태로 돌아가지 않습니다. 게다가 저장 전이라 새로고침하면 서버값으로 다시 복원되어 혼란이 가중됩니다. **수정:** 초기화 = 로드 기본값과 동일 규칙으로 통일하거나, 버튼명을 "전체 해제" 등 실동작에 맞게 변경. (테스트 P2-7)

### B · 현재 정책을 확인해야 판단할 수 있는 문제

**B1 · "인정 개수 30개"가 미적용 환경에서 하드코딩 표시**
인정 컬럼 미적용/미계산이면 항상 `DEFAULT_WEEK_RECOGNITION_COUNT=30`이 표시됩니다. 실제 성공 기준과 다를 수 있으나 화면상 구분이 없습니다. **확인 필요:** 미산정 시 "30" 대신 "미산정" 또는 별도 표기가 정책상 맞는가. (P1-4)

**B2 · 오픈 확인 완료 후에도 버튼 상시 활성 · 동기 중복 가드 부재**
오픈 확인 후에도 버튼이 비활성화되지 않아 재클릭 시 config 재저장+N 재계산이 반복됩니다(현재/미래 주차는 무경고). 검수 버튼은 동기 `ref` 가드가 있으나 오픈 확인은 없어 이중 클릭 시 중복 POST 가능(멱등이라 파손은 없음). **확인 필요:** 완료 후 재확인 허용 범위·중복 가드 필요성. (P2-1)

**B3 · "라인(개설)" 체크박스와 목록 "라인칸 개설율"의 SoT 분리**
체크박스는 `config`(액트 가동 판정용)에 저장되고, 개설율은 실제 개설 라인 집계라 서로 연동되지 않습니다. 설계 의도이나 검증자가 혼동하기 쉽습니다. **확인 필요:** 두 개념의 화면 명칭/안내 분리 여부. (P4-5)

**B4 · 비적용 주차(레거시/공식휴식/현재·미래)는 UI로 검수 불가**
검수 준비 모달이 not-applicable이면 [검수 완료] 버튼이 영구 비활성입니다. 과거 레거시 주차엔 합리적이나, 정책상 의도된 완전 차단인지 확인이 필요합니다. (P2-2 연계)

**B5 · 오픈 확인이 검수의 하드 게이트임을 안내하는 문구**
오픈 확인 없이 검수하면 준비 모달의 "오픈 확인 및 주차 성공 기준"이 ❌ 처리되고 서버 422로 막힙니다(단계 강제는 바람직). **확인 필요:** 첫 사용자가 "왜 검수가 안 되는지"를 모달만으로 충분히 이해하는지. (P2-2)

### C · 정상 동작이지만 사용성 개선을 고려할 수 있는 문제

**C1 · 오픈 확인 전 액트/라인 카드가 전부 회색**
비가동 상태(정상)이지만 "오픈 확인 후 가동됩니다" 같은 안내가 없어 초행자가 고장으로 오인할 수 있습니다. (P1-3)

**C2 · "관리 주차" vs "현재 주차" 구분 모호**
두 배너가 시각적으로 유사해 어느 주차를 검수하는지 헷갈릴 수 있습니다. (P1-2)

**C3 · 미저장 변경에 대한 이탈 경고 없음**
허브 체크 후 오픈 확인 없이 이탈/새로고침하면 무경고로 유실됩니다. (P2-6)

**C4 · 검수 부분 실패의 조용한 진행**
코호트 snapshot 재계산은 best-effort라 일부 실패해도 응답은 성공입니다. 고객 route lazy 재계산이 보정하지만, 관리자에게 부분 실패 신호가 없습니다. (P2-3)

**C5 · 상단 허브 용어 ↔ 하단 탭 용어 연결이 암묵적**
"라인 급(체크)"→[액트 체크 관리], "라인(개설)"→[라인 개설 관리]의 대응이 이전 버전 안내문 제거로 암묵화되었습니다. 초행자에겐 매핑이 불명확할 수 있습니다.

**C6 · 라인 개설 관리 탭은 클릭해야 로드**
기본 탭이 액트라, 첫 진입 시 라인 개설 데이터는 탭 클릭 전까지 조회되지 않습니다(성능상 합리적). "데이터 없음"으로 오인할 여지만 점검.
